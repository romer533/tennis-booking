# tennis-booking — контекст проекта для Claude

## Что это

Сервис, который автоматически и с минимальной задержкой бронирует слоты на корты клуба **Daulet** в CRM **Altegio** (`b551098.alteg.io`). Слот на дату D открывается на бронирование ровно в **07:00 Asia/Atyrau (UTC+5)** в день D−3 и выкупается за секунды. Ручное бронирование через мобильное приложение Altegio проигрывает другим ботам — отсюда необходимость своего.

Цель: по заданному расписанию `(день_недели, время_слота, корт, профиль_клиента)` гарантированно занять слот в момент открытия окна.

## Главные инварианты (нельзя нарушать)

1. **Тайминг — основное конкурентное преимущество.** Каждые 100 мс задержки — потенциальный проигрыш слота. Все решения в коде взвешиваются через призму latency. NTP-синхронизация часов обязательна.
2. **Часовой пояс — `Asia/Atyrau` (UTC+5, без DST).** Никаких "наивных" `datetime` в коде расписания. Только timezone-aware. Любое преобразование `local ↔ UTC` — через `zoneinfo`.
3. **Окно открытия слота = T−3 суток ровно в 07:00 Atyrau.** Например, чтобы забронировать корт на пятницу 18:00, окно открывается во вторник 07:00. Эта формула не должна быть зашита в магических константах в нескольких местах — только в одном модуле scheduler.
4. **Phase 0 (исследование API) блокирует всю остальную разработку.** До тех пор пока в [docs/api-research.md](docs/api-research.md) не задокументирован полный booking flow со всеми эндпоинтами, заголовками, телами запросов и нюансами авторизации/CSRF — НЕ писать клиент, scheduler или engine. Любой код, написанный до этого, придётся выкинуть.
5. **Real money / реальные слоты в проде.** Любой запуск против `b551098.alteg.io` с боевыми параметрами — это реальная попытка занять слот, который иначе достался бы живому человеку. В dev/test использовать **dry-run режим** (формирует запрос, но не отправляет POST на создание брони) или фиктивные временные точки в прошлом.
6. **Никаких автоматических массовых попыток "перебором".** Сервис обслуживает явное расписание пользователя, а не сканирует все доступные слоты. Это намеренное ограничение: чтобы не превратиться в источник DoS или скальпер-бота.

## Стек (предлагаемый, можно пересмотреть до начала Phase 1)

- **Python 3.11+**, `asyncio`
- HTTP: `httpx` (async, поддержка HTTP/2, переиспользование соединений)
- Конфиги: `pydantic` v2 + YAML
- Логи: `structlog` в JSON
- Тесты: `pytest`, `pytest-asyncio`, `respx` (мок httpx)
- Управление зависимостями: `uv` или `poetry`

## Конвенции

- **Время в коде:** только `datetime` с `tzinfo`. Внутренний канонический формат — UTC. Преобразование в `Asia/Atyrau` — только на границах (парсинг расписания, форматирование логов для человека).
- **Сетевые вызовы:** все через единый клиент `AltegioClient`. Никаких `httpx.get(...)` напрямую из бизнес-логики.
- **Сетевая ошибка ≠ "слот не доступен".** Различать: 4xx бизнес-ошибка vs 5xx/timeout/connection error. Для последних — повторять в пределах окна попыток.
- **Идемпотентность.** При retry внутри окна возможно дублирование брони. Проверять, не выиграли ли уже.
- **Логи:** структурированные (JSON), обязательные поля: `attempt_id`, `slot_dt_local`, `court_id`, `phase` (`schedule`|`prearm`|`fire`|`result`), `latency_ms`, `http_status`.

## Чего НЕ делать

- Не вшивать `b551098` или `Atyrau` в коде в десяти местах — только константы в одном модуле.
- Не использовать `time.sleep()` в коде, чувствительном к latency. Только `asyncio.sleep()`.
- Не "оптимизировать" Phase 0 — пройди реальный booking flow вручную через DevTools и задокументируй ВСЁ, даже если кажется неважным (cookies, заголовки, последовательность запросов).
- Не писать ретраи "на всякий случай" вокруг каждого вызова. Ретраи — только в booking engine, с явной семантикой "пока не истекло окно или не получили бизнес-успех/бизнес-отказ".
- Не добавлять "красивые" абстракции (DI-контейнеры, плагины, хуки) до того как заработает базовый сценарий end-to-end.
- Не запускать боевой режим против Altegio из dev-окружения. Должен быть явный флаг `--live` и подтверждение.

## Ролевые skills (в `.claude/skills/`)

Проект имеет 5 ролевых skills, которые автоматически подтягиваются Claude по контексту запроса:

| Skill | Когда активируется |
|-------|-------------------|
| **product-owner** | "что с проектом", "стоит ли делать X", "почему проиграли слот", "когда релизим", разбор инцидентов, приоритизация |
| **backend-engineer** | Любая задача с кодом в `src/tennis_booking/` |
| **code-reviewer** | Перед открытием PR, перед merge, ревью диффа |
| **qa-engineer** | Воспроизведение бага, тест-план, написание регрессионного теста, валидация фикса |
| **altegio-research** | Phase 0 — разбор запросов из DevTools, заполнение `api-research.md` |

Отдельно subagent `api-researcher` в `.claude/agents/` — вызывается через `Task` для анализа HAR-файлов в изолированном контексте.
