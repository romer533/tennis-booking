# План реализации

Каждая фаза имеет чёткий **выход** (deliverable) и **критерий готовности** (DoD). Без выполнения DoD предыдущей фазы — не начинать следующую.

## Phase 0 — Исследование Altegio API (БЛОКИРУЮЩАЯ)

**Цель:** полностью понять, какие HTTP-запросы делает фронтенд `b551098.alteg.io` от открытия страницы до подтверждения брони.

**Что делать (вручную):**

1. Открыть Chrome → DevTools → Network (Preserve log = ON, Disable cache = ON).
2. Очистить cookies для домена.
3. Открыть `https://b551098.alteg.io` в анонимном режиме.
4. Пройти реальный пользовательский флоу до момента "слот забронирован":
   - выбор услуги/категории (теннис → корт)
   - выбор сотрудника (если применимо) или конкретного корта
   - выбор даты
   - выбор временного слота
   - ввод имени и телефона
   - подтверждение (возможно, SMS-код)
5. Для каждого XHR/fetch-запроса в флоу зафиксировать в [api-research.md](api-research.md) по шаблону [research-template.md](research-template.md):
   - метод, полный URL, query-параметры
   - все request headers (особое внимание: `Authorization`, `X-CSRF-Token`, `Cookie`, `User-Agent`, любые `X-*`)
   - тело запроса (JSON / form-data)
   - статус и тело ответа
   - какие cookies сервер выставляет
6. Дополнительно проверить:
   - есть ли явная авторизация / токен / partner_token (Altegio публикует [Booking Form API](https://developers.alteg.io/) — сравнить с тем, что видим)
   - есть ли rate-limiting (повторить запрос несколько раз быстро — отдаёт 429?)
   - есть ли captcha / hCaptcha / reCAPTCHA в любой точке флоу
   - что происходит, если слот уже занят (попробовать забронировать заведомо взятый слот)
   - что происходит, если время бронирования ещё не открыто (попробовать слот через 4 дня — сервер вернёт пустой список или ошибку?)
7. Зафиксировать "горячий путь" (минимальный набор запросов, нужный, чтобы создать бронь, если уже знаем `service_id` / `staff_id` / `datetime`). Это то, что Phase 4 будет исполнять в момент T−0.

**DoD:**
- `docs/api-research.md` содержит все запросы flow.
- Явно выписан "горячий путь" (1–3 запроса), которые нужно выстрелить в момент открытия слота.
- Понятно, нужна ли SMS-верификация и как с ней быть (если нужна — это меняет всю архитектуру; см. Phase 1.5).
- Понятно, есть ли rate-limit / captcha и при каких условиях срабатывает.

## Phase 1.5 — Решение по SMS / captcha (если применимо)

Активируется, только если Phase 0 показала, что без SMS-кода или captcha бронь не оформить.

Возможные ответы:
- **SMS-код:** нужен механизм его получения за секунды (виртуальный номер с API, либо реальная SIM + Android-bridge типа `gammu`/`adb`). Закладывает 5–30 секунд задержки → меняет тайминг-стратегию (отправлять *первый* запрос ДО открытия окна и держать "открытую сессию"). Решение по вендору SMS-приёма принимается до Phase 2.
- **Captcha:** проверить, действительно ли срабатывает на каждый запрос или только при подозрении (часто только при многократных fail). Если только на fail — попадать с первого раза. Если на каждый — нужен solving-сервис (2captcha и пр.) или браузерная автоматизация (`playwright`) — это серьёзно меняет архитектуру.

**DoD:** в [docs/architecture.md](architecture.md) задокументировано принятое решение и его влияние на Phase 4.

## Phase 2 — Altegio HTTP-клиент

**Цель:** типизированный async-клиент, оборачивающий все эндпоинты из Phase 0.

**Что делать:**
- `src/tennis_booking/altegio/client.py`: класс `AltegioClient(base_url, http_client)` с методами по числу эндпоинтов горячего пути + вспомогательные (получение списка кортов, доступных слотов на дату).
- Pydantic-модели запроса/ответа для каждого эндпоинта.
- Поддержка переиспользования соединений (`httpx.AsyncClient` с явным connection pool, HTTP/2 включить, если сервер поддерживает).
- Метод `prearm()` — открывает соединения, делает разогревочный GET (например, главная страница), чтобы к моменту T−0 TLS-сессия и DNS уже были установлены.
- Чёткое разделение ошибок: `AltegioBusinessError` (4xx с известным кодом) vs `AltegioTransportError` (5xx, network, timeout).
- Тесты на `respx` — ожидаемое тело запроса и парсинг ответа для каждого метода.

**DoD:** dry-run скрипт `scripts/manual_book.py --dry-run` формирует валидный запрос на бронь по фиктивному слоту в прошлом, печатает что бы отправил, не отправляет. С флагом `--live` против тестового профиля — бронирует реально.

## Phase 3 — Scheduler

**Цель:** компонент, который точно знает, *когда* нужно стрелять.

**Что делать:**
- `src/tennis_booking/scheduler/window.py`: чистая функция `next_open_window(slot_local_dt: datetime, now_utc: datetime) -> datetime` — возвращает UTC-время открытия окна (= slot_local_dt − 3 суток в Atyrau, в 07:00). Покрыть юнит-тестами на edge-cases (слоты на стыке суток).
- `src/tennis_booking/scheduler/clock.py`: обёртка над часами с проверкой NTP-дрифта (через системный `chrony`/`w32tm` либо отдельный SNTP-запрос к `time.cloudflare.com` при старте). Отказ запускаться, если дрифт > 50 мс.
- `src/tennis_booking/scheduler/loop.py`: главный цикл. Раз в сутки в 06:55 Atyrau пересчитывает все ближайшие окна на ближайшие 24 часа и регистрирует таски `asyncio.create_task` под каждое.
- За **30 секунд** до окна — таск просыпается, поднимает соединения через `client.prearm()`.
- За **1 секунду** до окна — таск входит в "tight loop", проверяя `monotonic_ns()`, чтобы выстрелить максимально близко к T−0 (но НЕ раньше, чтобы не получить отказ).

**DoD:** unit-тесты на `next_open_window` (включая 1 января, переходы суток, корт на воскресенье 23:30); integration-тест на `loop.py` с подменённым клоком — таск просыпается с точностью ≤ 5 мс.

## Phase 4 — Booking engine

**Цель:** склеить scheduler + client в попытку бронирования.

**Что делать:**
- `src/tennis_booking/engine/attempt.py`: `class BookingAttempt` — описывает одну запланированную попытку (slot, court, profile, окно).
- Стратегия выстрела:
  - **T−0:** отправить **N параллельных** POST-запросов на создание брони (N = 2–3, чтобы первый, кто долетит, выиграл). Если N > 1 — первый успех отменяет остальные (cancel задачи).
  - На бизнес-ошибку "слот ещё не открыт" — повторить через 100 мс. И так до T+5s.
  - На бизнес-ошибку "слот занят" — стоп, лог "lost".
  - На бизнес-успех — лог "won", запросить подтверждение/детали брони, остановить параллельные попытки.
  - На transport-ошибку — повторить немедленно.
  - Глобальный дедлайн: T+10s. После — "timeout".
- Idempotency: после первого успеха проверять список своих броней перед каждым новым POST, чтобы не задвоить.

**DoD:** end-to-end тест на `respx`: эмулирует "слот закрыт до T−0, открыт после, занят после T+2s" — engine должен бронировать успешно. Эмулирует "слот занят сразу" — engine должен зафиксировать loss за < 200 мс.

## Phase 5 — Конфигурация

**Цель:** удобный для пользователя способ задать "что и когда бронировать".

**Что делать:**
- `config/schedule.example.yaml`:
  ```yaml
  bookings:
    - name: "Пятница вечер, парный"
      weekday: friday
      slot_local_time: "18:00"
      duration_minutes: 60
      court_id: <из api-research>
      profile: roman
      enabled: true
  ```
- `config/profiles.example.yaml`:
  ```yaml
  profiles:
    roman:
      full_name: "..."
      phone: "+7700..."
      email: "..."
  ```
- Валидация через pydantic при старте — с понятными ошибками (не raw stacktrace).
- CLI: `python -m tennis_booking run --config config/`.

**DoD:** запуск с пустым / битым конфигом → понятное сообщение об ошибке, не traceback. Запуск с корректным конфигом → лог "scheduled N bookings for next 7 days".

## Phase 6 — Observability

**Цель:** в любой момент знать, что произошло.

**Что делать:**
- `structlog` JSON в stdout + ротация в файл.
- Каждая попытка логирует фазы: `scheduled`, `prearm_started`, `prearm_done`, `fire_at`, `response_received`, `result`.
- `latency_ms` для каждой фазы.
- Опционально: Telegram-уведомление "выиграл слот X" / "проиграл слот X" / "ошибка".
- Опционально: SQLite с таблицей `attempts` (для post-mortem и метрик "сколько раз выиграл").

**DoD:** после прогона за день видно, на сколько мс отстаёт первый POST от T−0, и какой процент попыток успешен.

## Phase 7 — Деплой и хардниг

**Цель:** работает 24/7, переживает рестарты.

**Что делать:**
- Systemd unit (или Docker) на сервере, географически близком к Altegio (вероятно Almaty / Yandex Cloud, нужно замерить ping до `b551098.alteg.io`).
- NTP синхронизация на сервере (`chrony`).
- Health check.
- Rotation логов.
- Резервный сервер (опционально, если критично).

**DoD:** сервис пережил рестарт сервера, продолжил расписание; ping до Altegio < 50 мс; ntpq показывает offset < 5 мс.
