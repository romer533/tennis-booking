# Архитектура

> Этот документ описывает **целевую** архитектуру MVP. Часть решений зависит от выводов Phase 0 — они помечены `(зависит от Phase 0)` и должны быть пересмотрены после её завершения.

## Высокоуровневая схема

```
                   +---------------------+
   schedule.yaml ─►│  ConfigLoader       │
   profiles.yaml ─►│  (pydantic)         │
                   +----------+----------+
                              │
                              ▼
                   +---------------------+        +-------------------+
                   │  Scheduler          │◄──────►│  Clock + NTP      │
                   │  (asyncio loop)     │        │  (drift check)    │
                   +----------+----------+        +-------------------+
                              │  spawns task per (slot, profile)
                              ▼
                   +---------------------+        +-------------------+
                   │  BookingEngine      │◄──────►│  AltegioClient    │
                   │  (one per attempt)  │        │  (httpx async)    │
                   +----------+----------+        +-------------------+
                              │
                              ▼
                   +---------------------+
                   │  Observability      │
                   │  (structlog, opt:   │
                   │   sqlite, telegram) │
                   +---------------------+
```

## Компоненты

### `config/`
Парсит YAML, валидирует через pydantic. Возвращает иммутабельные `Schedule` (список `BookingRule`) и `Profiles` (словарь `name → Profile`). Не имеет состояния.

### `scheduler/`
- **`window.py`** — чистая функция `next_open_window(slot_local_dt) -> datetime`. Реализует правило "T−3 суток в 07:00 Almaty", возвращает UTC. Валидация "окно ещё в будущем" — ответственность `loop.py`, не функции (single responsibility). Полностью покрыта unit-тестами.
- **`clock.py`** — обёртка с проверкой NTP-дрифта. На старте проверяет offset; падает, если > 50 мс.
- **`loop.py`** — главный цикл. Раз в сутки в 06:55 Atyrau пересчитывает все ближайшие окна на 24 часа вперёд. Для каждого окна `asyncio.create_task` с пробуждением за T−30s.

### `altegio/`
- **`client.py`** — `AltegioClient`. Один экземпляр на процесс, переиспользует `httpx.AsyncClient` (HTTP/2 если поддерживается, явный connection pool).
- **`models.py`** — pydantic-модели запросов/ответов.
- **`errors.py`** — `AltegioBusinessError` (4xx с известным кодом ошибки) vs `AltegioTransportError` (5xx, network, timeout). Это **критическое разграничение** — booking engine действует на них по-разному.
- **Метод `prearm()`** — поднимает соединения, прогревает TLS. Вызывается за 30 секунд до T−0.

### `engine/`
- **`attempt.py`** — `BookingAttempt`. Описывает одну попытку (slot, court, profile, окно).
- Логика выстрела:
  - `T−30s`: `client.prearm()`
  - `T−1s`: tight loop на `monotonic_ns()` для точности
  - `T−0`: запустить N (=2–3) параллельных POST. Использовать `asyncio.wait(..., return_when=FIRST_COMPLETED)`. Первый успех → отменить остальные.
  - `BusinessError "не открыто"`: повторить через 100 мс. Дедлайн на эту фазу: T+5s.
  - `BusinessError "занято"`: stop, log "lost".
  - `BusinessError "успех"`: stop, log "won".
  - `TransportError`: повторить немедленно. Глобальный дедлайн: T+10s.

### `obs/`
- structlog в JSON, ротация в файл.
- Опционально: Telegram-бот для уведомлений (для алертов, не для интерактива).
- Опционально: SQLite для истории попыток.

## Ключевые решения и обоснования

### Тайминг

**Решение:** Python + `asyncio` + `httpx`. Не Go, не Rust.

**Обоснование:** доминирующая компонента latency — RTT до сервера Altegio (десятки мс) и его собственное время обработки. Разница между Python `httpx` и Go `net/http` на одиночном POST — единицы мс, шум на фоне сети. Простота и скорость разработки выигрывают.

**Что делаем для минимизации latency, что важнее выбора языка:**
1. Хостинг географически близко (Almaty).
2. Pre-arm (TLS handshake, DNS, HTTP/2 connection) за 30 сек до T−0.
3. NTP синхронизация часов.
4. N параллельных запросов в T−0.

### N параллельных запросов

**Решение:** N = 2–3 (зависит от Phase 0 — нужно проверить, не банит ли Altegio за параллелизм).

**Обоснование:** дублирующие запросы покрывают единичные потери пакетов и микро-фризы JVM/GC на стороне сервера. Большее N — diminishing returns + риск срабатывания rate-limit.

### Pre-arm vs ленивый коннект

**Решение:** обязательный pre-arm за 30 сек.

**Обоснование:** TLS 1.3 handshake = 1 RTT. На холодном соединении это прибавит ~30–50 мс к первому запросу. На горячем — 0. В гонке за слот это решает.

### NTP

**Решение:** проверка дрифта на старте, отказ запускаться при > 50 мс.

**Обоснование:** если часы убегают/отстают, scheduler выстрелит не в T−0. Лучше упасть громко при старте, чем тихо проиграть слот.

### Сервер бронирования

**Решение:** хостить в Казахстане (Almaty) или ближайшем регионе. Зависит от того, где сам Altegio (нужно затрассировать).

**Обоснование:** RTT — главная составляющая latency. Каждые 10 мс RTT = 10 мс отставания на каждом запросе.

### SMS / Captcha (зависит от Phase 0)

См. [PLAN.md, Phase 1.5](PLAN.md). Решение принимается после исследования.

### Idempotency после первого успеха

**Решение:** перед отправкой второго (повторного) POST — лёгкий GET со списком своих броней. Если уже есть нужная — стоп.

**Обоснование:** retry внутри окна может задвоить бронь. Иногда Altegio может вернуть transport error на запрос, который реально был успешно обработан.

## Что осознанно НЕ делаем

- **Браузерная автоматизация (Playwright/Selenium).** Медленно, тяжело, ломается на обновлениях фронта. Идём через "сырой" HTTP. Если Phase 0 покажет, что без браузера никак (тяжёлый JS, antibot fingerprinting) — пересматриваем.
- **Очереди / брокеры (Redis, RabbitMQ).** Один сервис, одно расписание, никакой межпроцессной координации. Всё в `asyncio` в одном процессе.
- **Базу данных в core flow.** Конфиги — в файле, история — в опциональном SQLite. Никакой БД на пути попытки бронирования (latency).
- **Гибкий plugin-architecture.** Один CRM (Altegio), один клуб (Daulet). YAGNI.

## Риски и mitigation

| Риск | Mitigation |
|------|------------|
| Altegio меняет API → клиент ломается | Phase 0 повторяется, тесты на `respx` ловят несоответствие на CI |
| Altegio добавляет captcha / antibot | Падаем громко, переходим на Phase 1.5 |
| Сервер падает в 06:59:59 | Health-check + systemd auto-restart; идеально — 2 инстанса в active-active (отложено за MVP) |
| Сетевой провайдер моргнул в T−0 | N параллельных запросов помогают на единичных потерях. Системные сбои — вне MVP. |
| Часы убежали | NTP-проверка на старте + повторно перед каждым окном. |
| Слот занят живым человеком за миллисекунду до нас | Принимаем как фоновый риск. Логируем, ничего не делаем. |
