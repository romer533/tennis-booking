# Шаблон документирования API-вызовов Altegio

Этот файл — **шаблон**. По нему ведётся [api-research.md](api-research.md) во время Phase 0.

Для **каждого** наблюдённого XHR/fetch-запроса в букинг-флоу заполняется блок ниже. Не пропускай "очевидные" запросы (загрузка списка категорий, картинок и т.д.) — позже они могут оказаться обязательной разогревочной частью сессии.

---

## Запрос N: <короткое название, напр. "Получить список услуг">

**Когда происходит:** <напр. "сразу при открытии главной страницы">

**Зачем нужен:** <напр. "получить service_id корта, который потом передаётся в /book">

**Метод и URL:**
```
POST https://b551098.alteg.io/...
```

**Query-параметры:**
| Параметр | Значение | Откуда берётся |
|----------|----------|----------------|
|          |          |                |

**Request headers (значимые):**
```
Authorization: Bearer ...
X-CSRF-Token: ...
Cookie: session=...; csrftoken=...
Content-Type: application/json
User-Agent: ...
Origin: https://b551098.alteg.io
Referer: https://b551098.alteg.io/...
```
Пометить, какие из них:
- статичные (одинаковые между запросами)
- динамические (меняются — указать, откуда берутся: предыдущий ответ / cookie / JS-генерация)

**Request body:**
```json
{
  "...": "..."
}
```

Для каждого поля — комментарий, откуда взялось значение (захардкожено фронтом / из формы / из ответа предыдущего запроса).

**Response (success):**
- статус: `200`
- тело:
```json
{
  "...": "..."
}
```
- какие cookies сервер выставил
- какие поля ответа критичны для следующих запросов (выписать явно)

**Response (errors), которые удалось спровоцировать:**
- `400` при <условие> — тело: `{...}`
- `409` при попытке забронировать занятый слот — тело: `{...}`
- `404` при попытке получить недоступный/не открытый слот — тело: `{...}`

**Заметки:**
- Идемпотентен ли? (повторный вызов с тем же телом — что вернёт?)
- Есть ли rate-limit? (спам N запросов в секунду — на каком N начинается 429?)
- Зависит ли от cookie/сессии, заведённой предыдущим запросом?

## Сводка после прохождения флоу

### Горячий путь (что нужно стрелять в T−0)

Минимальная последовательность запросов, которая создаёт бронь, при условии что `service_id`, `staff_id`, `slot_datetime`, `customer.name`, `customer.phone` уже известны:

1. <запрос 1> — обязателен / опционален?
2. <запрос 2>
3. ...

Какие cookies/токены нужно иметь к моменту T−0? Можно ли их получить заранее (за минуту до окна) и переиспользовать?

### Авторизация / CSRF

- Требуется ли auth-токен? Если да — как он получается, какой TTL?
- Есть ли CSRF? Если да — где приходит, как используется?
- Что произойдёт, если не передать какой-нибудь "необязательный" заголовок (`Origin`, `Referer`)?

### Подтверждение по SMS / Captcha

- Есть ли SMS-код в флоу? На каком шаге? Можно ли пропустить?
- Есть ли captcha? Когда срабатывает (всегда / при подозрении)?

### Поведение на закрытом слоте (T < открытие окна)

- Что вернёт API, если попытаться забронировать слот, до открытия которого ещё 1 минута?
  - 404? Пустой список? Бизнес-ошибка с конкретным кодом?
- Это критично для стратегии retry в booking engine.

### Rate-limiting

- Какие лимиты обнаружены? IP-based или session-based?
- Влияет ли на выбор стратегии "N параллельных запросов в T−0"?

### Прочие наблюдения

- Версия API в URL?
- WebSocket-соединения в флоу?
- Что-то странное / неочевидное?
