# Методология исследования Altegio API

Это пошаговый чек-лист для Phase 0 проекта tennis-booking. Цель — собрать минимально достаточный, воспроизводимый набор HTTP-вызовов для бронирования слота на `b551098.alteg.io`.

## Подготовка

1. Открыть Chrome → новое окно в режиме инкогнито.
2. DevTools (F12) → вкладка Network:
   - **Preserve log** — ON
   - **Disable cache** — ON
   - **Fetch/XHR** фильтр (картинки и шрифты на этом этапе не нужны)
3. В отдельной вкладке открыть [docs/research-template.md](../../../docs/research-template.md) — будет шаблон для каждого запроса.
4. Целевой файл результата: [docs/api-research.md](../../../docs/api-research.md).

## Основной флоу (записывать каждый шаг)

Пройти **реальное** бронирование слота, который не жалко занять/отменить:

1. Открыть `https://b551098.alteg.io`. Зафиксировать первые запросы (какие cookies сервер выставляет, какие GET к API он делает).
2. Выбрать категорию услуги (теннис).
3. Выбрать конкретный корт (или сотрудника, в зависимости от UX).
4. Выбрать дату.
5. Выбрать время.
6. Ввести имя и телефон.
7. Подтвердить бронь (ввести SMS-код, если попросят).

После каждого клика — пауза, чтобы понять, какие запросы относятся к какому действию. Полезно делать пометки прямо в Network tab (правый клик → "Save HAR with content" в конце для архива).

## Что записывать про каждый запрос

Использовать [research-template.md](../../../docs/research-template.md). Минимум:

- метод + полный URL (включая query string)
- request headers — все, кроме браузерных дефолтов (`Accept-Language`, `sec-ch-*` и т.п.). Особое внимание: `Authorization`, `X-CSRF-*`, `Cookie`, `Origin`, `Referer`.
- request body
- response status + body
- какие cookies сервер выставил/обновил

Для каждого динамического значения (токен, id, csrf) — указать, **откуда оно взялось**: предыдущий ответ / cookie / сгенерировано фронтом.

## Обязательные провокации

После прохождения happy path спровоцировать ошибки — это часто важнее самого happy path:

1. **Слот занят.** Попробовать забронировать слот, который уже занят. Какой статус, какое тело ответа?
2. **Слот ещё не открыт.** Через `service` API запросить слот через 4 дня (окно открывается за 3). Что вернёт — пустой список, 404, бизнес-ошибка с кодом? Это критично для retry-стратегии в booking engine.
3. **Дубликат.** Сразу после успешной брони повторить тот же POST. Идемпотентен ли? Создаст вторую бронь? Вернёт ошибку?
4. **Без cookies/headers.** Повторить успешный POST через cURL без cookies, без `Origin`, без `Referer`. Что обязательно? (Это определяет, можно ли стрелять "холодно", без разогревочного GET.)
5. **Параллелизм.** Запустить из терминала 3–5 одновременных одинаковых POST на (другой, не нужный никому) слот. Получим 429? Один пройдёт, остальные отвергнутся? Все пройдут (создадут дубликаты)? Это ключ к стратегии "N параллельных запросов в T−0".

## Сводка

В конце обязательно заполнить блок "Сводка" в `api-research.md`:

- **Горячий путь:** какие 1–3 запроса нужны, чтобы создать бронь, при условии что `service_id`/`staff_id`/`datetime`/`name`/`phone` уже известны.
- **Авторизация:** есть ли токен/CSRF, как получается, можно ли разогреть заранее.
- **SMS / captcha:** да/нет, в каком шаге.
- **Поведение на закрытом слоте:** что вернёт сервер.
- **Rate-limit:** при каком N параллельных запросов начинаются отказы.

## Гейт перед закрытием Phase 0

Phase 0 закрывается, только если:

- [ ] Сводка заполнена и однозначна (а не "вероятно", "наверное").
- [ ] Все запросы happy path задокументированы.
- [ ] Все 5 провокаций проведены, результаты записаны.
- [ ] HAR-файл (`flow.har`) сохранён локально как референс (не коммитим — может содержать PII в виде моего телефона).

## Как использовать subagent api-researcher

Для разбора собранных данных без написания кода — звать subagent `api-researcher`. Он не пишет production-код, только анализирует и заполняет `api-research.md`.
